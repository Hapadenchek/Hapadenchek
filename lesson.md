# Lesson 7. Тестирование проекта по требованиям
* Повторите весь изученный материал
* Узнаете, что такое требования к программному обеспечению и их критерии качества
* Узнаете, как тестировать требования к программному обеспечению

```
Требования к программному обеспечению - описание свойств и качеств программного обеспечения, которыми оно должно обладать.
```
**Что такое требования к программмному обеспечению**
1. Бизнес-требования описывают для кого нужен продукт, какую проблему или проблемы решает.
2. Пользовательские требования описывают непосредственные задачи польщователя, которые он хочет решать,а также говорят о том кто наши пользователи.
3. Функциональные требования детально описывают поведение программного обеспенчения.
4. Нефункциональные требования описывают характеристики прогрммного обеспечения.

**Критерии качества требований к ПО**
1. Однозначность Требования не должны допускать несколько трактовок после прочтения.
2. Непротиворечивость Требования не должны противаоречить сами себе или другим пунктам требований.
3. Выполнимость Требование должно быть возможно реализовать технически,а также за отведенные на это сроки.
4. Проверяемость Реализацию требования можно проверить.
5. Обязательность Требованя должно быть обязательным к выполнению.

**Как тестировать требования к ПО**
1. Аналитик пишет требования и во время или после их написания презентует команде разработки.
2. Тестировщики и разработчики читают требования.
3. Финализированные требования аналитик фиксирует в документе и, при необходимости, дополнительно согласовывает с заказзчиком.

# Lesson 8. Классы эквивалентности и граничные значения
* Что такое тест-дизайн и зачем он вообще нужен
* Что такое классы эквивалентности и как их применять
* Что такое метод граничных значений и как его применять

1. Тест-дизайн это проектирование и создание тестовых сценариев, 
использую техники дизайна.
2. Техника тест-дизайна есть различные техники тест дизайна, чаще всего используются такие техники:
* Классы эквивалентности
* Граничные значение

**Граничные значение** – это значения, в которых один класс эквивалентности переходит в другой. По своей сути это техника, которая дополняет технику классов эквивалентности.

**Эквивалентности** Они тестируют одну и ту же вещь (функцию, модуль, часть системы).
Если один из тестов ловит ошибку, то другой скорее всего тоже её поймает.
Если один из них не ловит ошибку, то другой скорее всего тоже не поймает
Если же мы говорим об эквивалентности входных параметров программы, то входные параметры, которые приводят к одинаковому поведению программы, мы будем считать эквивалентными.

**Регрессионое тестирование** это тестирование системы, которое проводится после исправление багов или реализации новой функциональностей.

 **Проводится оно в след.случаях:**
 * исправление бага
 * реализована новая функциональность
 * готовится новый релиз приложения 

**Релиз новой версии**. Перед релизом новой версии приложеня проводить регрессионное тестирование нужно в несколько **прогонов**.

* Дальше мы начинаем первый **тестовый прогон** обычно их необходимо сделать несколько.

**Что такое API** это (Application Prigramming Interface)
программный интерфейс приложения, с помощью которого одна программа может взаимодействовать с другой. API позволяет слать информацию напрямую из одной программы в другую, миную интерфейс взаимодействие с пользователем.

**REST API** использует протокол HTTP и его стандартные методы.

**SOAP API** веб-сервис, использующий протокол SOAP для обмена сообщениями между серверами и клиентами. При этом сообщения должно быть написаны на языке XML в соответствии со строгими стандартами, иначе сервер вернет ошибку.

**http-методы**
* GET- предназначен для получение ресурса с сервера.
* POST- отправляет данные на сервер с созданием новой записи.
* PUT- отправляет данные на сервер с перезаписью существующей записи.
* DELETE- удаляет данные ресурса.
